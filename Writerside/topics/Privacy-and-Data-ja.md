# プライバシーとデータ

%product% を使用するときに何がマシンから送信され、どこに送られるのか、そしてすべてをローカルに保つ方法について説明します。

## 資格情報

| 資格情報                 | 保存場所                                                                                                              |
|--------------------------|---------------------------------------------------------------------------------------------------------------------|
| Azure DevOps PAT         | IDE の `PasswordSafe` → システムキーチェーン（macOS Keychain、Windows Credential Manager、GNOME Keyring / KWallet）。 |
| OAuth リフレッシュトークン | 同上 - `PasswordSafe`。ディスク上での平文保存は一切行われません。                                                    |
| AI プロバイダー API キー | 同上 - `PasswordSafe`。プロバイダーインスタンスごとにキー管理されます。                                              |

OS ごとのキーチェーンの詳細、および PAT のローテーションや失効の方法については、[認証](Authentication-ja.md)を参照してください。

## Azure DevOps に送信される内容

このプラグインは Azure DevOps REST API 上の薄いクライアントです。すべての呼び出しは、設定した組織（`dev.azure.com/<org>` またはオンプレミスの Azure DevOps Server）に直接送られます。サードパーティのサーバーを経由することはありません。

呼び出しが行われるのは次の場合です:

- PR ツールウィンドウを開いたとき（初回のリスト取得）。
- 60 秒ごとのバックグラウンド同期が実行されたとき。
- プルリクエストを開いたとき（タイムライン + 差分の取得）。
- コメントの投稿、投票、既読マーク、完了、または破棄を行ったとき。
- Azure DevOps リモートに対する `git fetch` / `git push` の資格情報受け渡しを生成したとき。

## AI プロバイダーに送信される内容 {id="whats-sent-to-ai-providers"}

AI が**有効**でプロバイダーが構成されている場合のみ送信されます。プラグインは、あなたがトリガーしたアクションに対してのみ外部への呼び出しを行います。

> AI プロバイダーを追加または編集すると、プラグインはその API キーを使用して、そのプロバイダーに対して認証済みの**モデルリスト**リクエスト（例: `GET /v1/models`）を 1 回行い、モデルのドロップダウンを埋めます。これは、トリガーしたアクションではなくセットアップ時に発行される唯一の AI リクエストです。PR のコード、差分、プロンプトは送信されず、送られるのはモデルリストの呼び出しだけで、結果は約 30 分間キャッシュされます。ローカルプロバイダー（localhost アドレスの Ollama）は、これをあなたのマシン内に留めます。
> {style="note"}

### 機能ごとのデータフロー

| 機能                   | プロバイダーが参照する内容                                                                                       |
|------------------------|-----------------------------------------------------------------------------------------------------------------|
| **Summarize PR**       | PR のタイトル + 説明 + 差分（**Max diff size**、既定 200 KB に切り詰め）。                                       |
| **AI review pass**     | 変更された各ファイルのファイル単位の完全な差分。**Max diff size** を超えるファイルはスキップされます。           |
| **Explain code**       | 選択したファイルの内容（表示範囲だけでなくファイル全体）。                                                       |
| **Commit message**     | ステージング済みの差分（`git diff --cached` が生成するもの）。                                                   |
| **Title & description**| ブランチ名 + コミットメッセージ + 差分（上記のとおり切り詰め）。                                                 |

各 AI リクエストには、プラグインがその機能向けに構築するシステムプロンプトも含まれます。これらのプロンプトは <ui-path>Settings | Version Control | Azure DevOps | AI Settings | Configure Prompts</ui-path> で上書きできます。

### プロバイダーのデータフロー一覧

| プロバイダー             | リクエストの送信先                                                                                                                             |
|--------------------------|------------------------------------------------------------------------------------------------------------------------------------------------|
| **Claude (Anthropic)**   | `api.anthropic.com`（またはカスタムのベース URL）。                                                                                            |
| **OpenAI**               | 既定では `api.openai.com`、または構成したベース URL（Azure OpenAI、vLLM、セルフホスト）。                                                      |
| **Gemini (Google)**      | Google の AI Studio / Vertex AI のエンドポイント。                                                                                             |
| **Ollama**               | 設定したエンドポイント、通常は `http://localhost:11434`。これが localhost アドレスの場合、**ネットワークへの送信はありません**。               |
| **Claude Code CLI**      | `claude` バイナリが認証とルーティングを処理します。データフローは Anthropic の CLI 規約によって制御されます。                                  |
| **OpenAI Codex CLI**     | `codex` バイナリの認証とルーティング。                                                                                                         |
| **GitHub Copilot CLI**   | `copilot` バイナリはあなたの Copilot サブスクリプションを使用します。データフローは GitHub Copilot の規約に準拠します。                        |

プラグインは、これらのリクエストにヘッダー、テレメトリ、分析を追加しません。上流のプロバイダーが参照する内容は、プラグインが送信した内容とまったく同じです。

## AI レスポンスのキャッシュ {id="caching-ai-responses"}

機能が結果を返すと、プラグインは次のキーでキャッシュします:

- PR ID。
- リクエスト時点の差分の SHA。
- 単調増加する `cacheGeneration` カウンター - プロンプトを編集したり、プロバイダーを変更したり、サマリーカードの歯車ポップアップで **Clear cached AI responses** をクリックすると加算されます。

キャッシュヒットの場合、外部への呼び出しなしで即座に結果が返されます。キャッシュされたレスポンスは IDE ローカルの状態に保存され、プラグインのアンインストール時にクリアされます。

## テレメトリ

出荷版のプラグインは**テレメトリを収集しません**。上記に列挙した API 呼び出し（あなたの Azure DevOps 組織と選択した AI プロバイダー宛て）以外に、使用状況の分析、クラッシュレポート、「phone home」ping は一切ありません。

将来のバージョンでオプトインのテレメトリが追加される場合は、既定でオフになり、リリースノートで明示されます。

## すべてをローカルに保つ

データレジデンシー要件のある組織向け:

1. **オンプレミスの Azure DevOps Server**（旧称 TFS）を使用する。プルリクエスト、コメント、API のすべてが自社サーバー上に存在します。
2. マスタースイッチで **AI を無効化する**、または `localhost` で稼働する **Ollama インスタンスにすべての AI 機能をルーティングする**。オフライン対応のモデル（Llama、Mistral など）と組み合わせれば、データはマシンから一切外に出ません。
3. サードパーティ CLI の規約を引き継ぎたくない場合は、**CLI 系の AI プロバイダーを避ける**。便利ですが、そのデータフローはプラグインからは不透明です。

## カスタム AI プロバイダー（エンタープライズ） {id="custom-ai-providers-enterprise"}

%product% は拡張ポイントを公開しており、エンタープライズの社内プラグインが組み込みの AI 実装を置き換え、たとえば社内ゲートウェイ経由で AI 呼び出しをルーティングできます。拡張ポイントは `plugin.xml` の名前空間 `intellij.vcs.azuredevops` の下で宣言されています。実装の詳細については、プラグインの GitHub リポジトリを参照してください。

より優先度の高い拡張が登録されている場合、その機能についてはプラグインの組み込みの既定が回避されます。
